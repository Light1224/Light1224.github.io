GEMM v4: NEON 4x4 and 8x8 Microkernels

Version 4 introduces architecture-specific SIMD kernels in gemm/v4_neon_4x4.cpp and gemm/v4_neon_8x8.cpp. Both paths compute full tiles using NEON vector registers and vfmaq_f32, with scalar fallback for non-full edge tiles.

4x4 kernel details

  • Four output rows each hold one 4-lane accumulator vector.
  • For each K step, one B vector is loaded and A scalars are broadcast.
  • Fused multiply-add updates c0..c3 vectors per iteration.
  • Final vectors are stored contiguously to C row segments.

8x8 kernel details

  • Each output row holds two 4-lane vectors (columns 0-3 and 4-7).
  • B loads are split into b0 and b1 per K step.
  • Eight A scalars are broadcast for row-wise accumulation.
  • Store phase writes 64 outputs after full K accumulation.

Edge and correctness handling

Both kernels guard full-tile execution with im/jm checks. If a tile is partial, control drops to a scalar cleanup path that computes exact results without assuming vector-width alignment. This keeps correctness independent of shape divisibility.

Performance implications

v4 is the first stage where compute throughput is no longer purely scalar-bound. In the run dump, v4_8x8 reaches substantially higher GFLOP/s than scalar versions on larger square and structured shape families. However, because v4 still consumes source layout directly, memory-side inefficiencies become the next bottleneck once compute gets faster.

Why v4 splits 4x4 and 8x8

The two kernels are useful for different scale/overhead regimes. The 4x4 path carries lower tile setup overhead and helps small shapes, while 8x8 has better compute density for larger blocks. v7 later uses this distinction in dispatch policy.
