GEMM v6: Parallel Packed Execution

Version 6 (gemm/v6_parallel.cpp) parallelizes the packed kernel while preserving the same fundamental compute building blocks (A/B packing, 8x8 NEON microkernel, scalar edge cleanup). The major additions are scheduling policy, persistent worker threads, and thread-local workspace reuse.

Parallel runtime architecture

  • A singleton ThreadPool persists worker threads across calls.
  • Each call submits a function and waits on pending_ completion.
  • Workers synchronize via generation counters and condition variables.
  • Per-thread memory is held in thread_local std::unique_ptr<Workspace>.

Scheduling policy and tile partitioning

v6 computes tiles_m and tiles_n from selected BM/BN. It chooses the parallel axis by larger tile count (parallel_over_n = tiles_n >= tiles_m), then caps thread count by available parallel tiles to avoid oversubscription. It also adjusts BM/BN downward for moderate problem sizes to reduce under-filled tiles.

Reuse patterns in both partition modes

  • When partitioning over N tiles, each worker packs B once per (kk, jj) and reuses across M tiles.
  • When partitioning over M tiles, each worker packs A once per (ii, kk) and reuses across N tiles.
  • Both paths then execute L1-blocked micro-tile loops with the same 8x8 kernel.

Robustness and controls

  • C is explicitly zeroed at kernel entry.
  • Thread count defaults to hardware concurrency and can be pinned by ATLAS_NUM_THREADS.
  • Boundary tiles keep scalar fallback, ensuring correctness for non-multiples of 8.

Observed benchmark profile

In the run dump, v6 is strongest on larger matrices and high-work scenarios where thread and packing overhead is amortized, while small matrices still show overhead sensitivity. This is consistent with the dispatch rules later codified in v7.
