GEMM v1: Loop Reorder (i-k-j)

Version 1 (gemm/v1_loop_reorder.cpp) keeps scalar arithmetic but changes the accumulation order from i-j-k to i-k-j. This is the first structural optimization in the repository and is intentionally small in code size so its effect can be reasoned about in isolation.

Code-level behavior

  • C is zero-initialized before accumulation.
  • For each (i, k), A[i, k] is loaded once into scalar a.
  • b points to row k of B, c points to row i of C.
  • Inner j loop performs contiguous updates c[j] += a * b[j].

Why this helps

In v0, each output element recomputes its entire dot product in isolation. In v1, each A[i, k] value is amortized over all columns of the current row of C. This improves effective reuse without introducing blocking state or auxiliary buffers.

Tradeoffs and constraints

  • Still scalar and single-threaded.
  • No explicit cache tile control across M/N/K dimensions.
  • Benefit depends on matrix shape and memory hierarchy behavior.
  • Code remains straightforward enough for direct debugging.

Validation and benchmark context

tests/benchmark_gemm_v1.cpp runs v1 over square, tall-skinny, wide, large-K, and cache-stress scenarios using shared timing logic. In the captured run dump, v1 shows a large step over v0 while retaining algorithmic simplicity, which is exactly the expected outcome for a loop-order-only change.

Role in progression

v1 demonstrates that data traversal order alone can shift performance substantially. It sets up v2, where the same accumulation pattern is lifted into blocked traversal to control working set size explicitly.
