GEMM v2: Cache-Blocked Kernel

Version 2 (gemm/v2_blocked.cpp) introduces explicit macro blocking using BM, BN, and BK from atlas_memory/include/atlas_memory/config_m2.hpp. It preserves scalar inner math, but constrains traversal so a bounded tile of A/B/C is reused before moving on.

Blocking strategy in code

  • Outer tile loops iterate ii, kk, then jj.
  • Tile boundaries use std::min for remainder-safe edges.
  • Inner body remains a * b + c scalar updates.
  • C is zeroed before blocked accumulation begins.

Memory behavior differences from v1

v1 improves reuse at the row level, but still sweeps large address ranges as dimensions grow. v2 bounds active subproblems by block sizes, reducing capacity and conflict pressure in cache. This does not change arithmetic complexity, but it changes how often the same lines are reused before eviction.

Design constraints

  • Block sizes are static constants, not runtime autotuned.
  • No packed buffers yet; source layout still drives loads.
  • No SIMD microkernel; scalar throughput remains a cap.
  • Control flow complexity increases versus v1.

Validation and measurement

tests/benchmark_gemm_v2.cpp applies the same scenario families as v1. The dump confirms v2 tracks v1 closely on some regions and improves stability on larger regimes where blocked reuse matters more. Correctness remains covered by shared GEMM tests and blocked-path checks such as test_basic_blocked_gemm.

Transition significance

v2 is the first version that reflects production GEMM structure: explicit macro tiling and predictable subproblem boundaries. v3 builds directly on this shape by introducing micro-tiles inside each block.
