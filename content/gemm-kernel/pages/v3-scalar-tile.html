GEMM v3: Scalar Micro-Tiling

Version 3 (gemm/v3_scalar_tile.cpp) keeps v2 macro blocks and adds micro-tiling at MR=4, NR=4. Instead of updating C for every inner multiplication, the kernel accumulates into a local stack tile (float acc[MR][NR]) and commits once per micro-tile.

Execution model

  • Macro loops remain blocked by BM/BN/BK.
  • Within each block, loops step by MR and NR.
  • Partial edges are handled via im and jm bounds.
  • Only after the K sweep completes is acc added into C.

Why micro-tiling matters before SIMD

Even in scalar code, local accumulation changes write behavior and shortens repeated access distance to the same sub-block of operands. This version is a structural bridge toward SIMD kernels because it introduces the notion of a fixed compute tile and explicit edge cleanup.

Practical tradeoffs

  • More control flow and temporary state than v2.
  • Still limited by scalar arithmetic width.
  • Tile shape is fixed in source (4x4), not adaptive.
  • Edge handling complexity increases but remains explicit and readable.

Observed behavior in benchmark suite

tests/benchmark_gemm_v3.cpp uses the full scenario matrix, and the recorded dump shows v3 generally above v0 but below NEON and packed kernels for large problems. This is expected: v3 improves execution structure but does not yet use vector FMA.

What v3 enables for v4

The move from scalar micro-tiles to NEON microkernels is then mostly a change in per-tile compute implementation. The iteration scaffolding and edge concepts introduced in v3 carry directly into v4 design.
