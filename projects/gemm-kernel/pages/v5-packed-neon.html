GEMM v5: Packed Panels + 8x8 NEON

Version 5 (gemm/v5_packed.cpp) is the first fully integrated packed kernel: macro tiles, L1 sub-blocking, MR-interleaved A packing, B panel packing, and an unrolled 8x8 NEON microkernel. This is where data layout is intentionally transformed to match compute.

Packing and workspace model

The kernel allocates one atlas_memory::Workspace per call. The workspace layout is deterministic and aligned: A region, B region, and accumulator region computed by atlas_memory/src/layout.cpp. Allocation uses posix_memalign at SIMD_ALIGNMENT=128, followed by page pre-touch in atlas_memory/src/workspace.cpp.

Compute path details

  • A is packed with pack_A_interleaved as [panel][k][i] (MR-interleaved).
  • B is packed row-panel-wise by pack_B.
  • Macro loops iterate ii/jj/kk; micro loops iterate i/j by MR/NR.
  • L1 blocking uses L1_BM and L1_BN to bound the inner working set.
  • Full 8x8 tiles run NEON; edges fall back to scalar cleanup with optional prefetch.

Kernel-level nuances

The 8x8 microkernel in v5 loads current C tile into registers, executes a 4-way unrolled K loop where available, and stores back once. Because A and B are pre-arranged for this access pattern, the hot loop avoids expensive gather-like behavior and relies on mostly contiguous reads.

Tradeoffs

  • Extra packing overhead on tiny matrices can dominate.
  • Code complexity rises due to packed indexing and dual-level blocking.
  • Throughput improves strongly once packing cost is amortized.
  • Single-threaded path keeps behavior deterministic and easier to profile.

Validation context

tests/benchmark_gemm_v5.cpp and benchmark_naive_vs_packed quantify the packed path, while test_packing_correctness, test_layout_and_alignment, and test_reset_behavior protect memory invariants that v5 depends on.
