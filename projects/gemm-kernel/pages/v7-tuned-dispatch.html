GEMM v7: Tuned Dispatcher and Policy Layer

Version 7 (gemm/v7_tuned.cpp) is a shape-aware policy layer that routes GEMM calls to v4, v5, or v6. Instead of introducing another compute kernel, it optimizes end-to-end behavior by selecting the right existing path for each regime.

Dispatch stages in the current implementation

  • If ATLAS_FORCE_KERNEL is set to v4, v5, or v6, forced dispatch is used.
  • max(M,N,K) <= 64 routes to gemm_v4_neon_4x4.
  • max(M,N,K) <= 192 routes to gemm_v4_neon_8x8.
  • Skinny shapes (min(M,N) <= 128, max <= 2048) route to v5.
  • Mid/large logic promotes to v6 when work and K-depth justify parallel overhead.
  • Fallbacks prefer v5 when macro tile fit is good, otherwise v4 8x8.

Policy intent

The dispatcher encodes a practical observation from the benchmark suite: there is no single best kernel across all shapes. Small matrices prefer low-overhead SIMD paths, skinny problems often favor packed single-thread execution, and larger dense workloads gain from v6 parallelism. v7 centralizes this choice so callers do not need to maintain per-shape logic.

Relationship to preceding versions

  • v4 contributes low-overhead 4x4/8x8 SIMD kernels.
  • v5 contributes robust packed single-thread throughput.
  • v6 contributes thread-level scaling and tile scheduling.
  • v7 provides policy cohesion and override control.

Validation and runtime observability

tests/benchmark_gemm_v7.cpp measures dispatcher behavior over the same five scenario families used by prior versions, and tests/benchmark_latency.cpp reports distribution metrics (min/median/p90/p95/p99/p999/mean/stddev/max) on fixed-size repeated runs. Together these capture both throughput and tail behavior under v7 policy decisions.

Current limitations and tuning surface

  • Thresholds are static and hand-tuned.
  • No automatic retuning based on hardware counters or deployment profile.
  • Policy can still be overridden for controlled experiments and A/B comparisons.
