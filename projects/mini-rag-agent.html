Mini RAG Agent

Mini RAG Agent is a minimal, end-to-end implementation of a Retrieval-Augmented Generation (RAG) system built to make every stage of the pipeline explicit, inspectable, and persistent.

Motivation

Modern RAG systems are often hidden behind large frameworks that abstract away core ideas such as chunking, retrieval, vector storage, and grounding. This project was built to understand those mechanisms from first principles by implementing a complete RAG pipeline with no heavy abstractions and no black-box orchestration.

System Architecture

The system is organized into a small number of clearly separated components, each with a single responsibility. The diagram below shows the high-level data flow for both document ingestion and query execution.

Frontend

The frontend is a lightweight browser interface implemented using plain HTML, CSS, and JavaScript. Its role is limited to user interaction: uploading text documents, submitting questions, and displaying responses. All application logic and state management reside on the backend.

Backend Runtime

The backend is implemented as a FastAPI application served via Uvicorn. Uvicorn provides the ASGI runtime responsible for request handling and asynchronous execution, while FastAPI defines the application layer that orchestrates ingestion, retrieval, and model interaction.

The backend itself is stateless. Each request is handled independently, and all persistent data is stored in the database. This design keeps execution deterministic and restart-safe.

Persistent Storage

PostgreSQL is used as the system’s persistent storage layer. It stores uploaded documents, chunked text segments, vector embeddings, and metadata linking chunks to their source documents.

Instead of relying on a specialized vector database, PostgreSQL was chosen to keep the storage model explicit and easy to inspect. This prioritizes clarity and correctness over scalability, which aligns with the project’s educational goals.

Model Runtime

Ollama is used as a local model runtime and serves two roles within the system. First, it generates embeddings for both document chunks and user queries. Second, it generates grounded answers using retrieved context and the user’s question.

Ollama is treated as a pure computation service with no hidden state. All orchestration, prompt construction, and retrieval logic remain within the backend.

Execution Flow

The system operates in two primary phases. During document ingestion, uploaded text files are chunked, embedded, and stored persistently. During query execution, the user’s question is embedded, relevant chunks are retrieved via vector similarity, and a grounded response is generated using the retrieved context.

There is no conversational memory; each query is processed independently to ensure predictable behavior.

Limitations

  • Supports plain text documents only
  • No advanced retrieval techniques such as reranking or hybrid search
  • No distributed or scalable vector storage
  • No authentication, observability, or production hardening

Status

Completed as a learning-focused system and reference implementation.
